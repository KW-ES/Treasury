Summary

Background & Current State

  • 95% of positions currently come from Murex, with the remaining 5% from other systems
  • Non-Murex positions include:
    • IBD trades (managed in ACPS)
    • Physical commodities (from Osiris)
    • Corporate bank trades (loans and deposits)
    • Operational call deposits and finance adjustments (from Smart/Chameleon)
  • Current risk runs are designed by entity sets (e.g., SA entity set)

Pass-Cash Problem

  • Position state gets imported into Murex
  • Market value component is used in attribution process
  • Cash balance comes from Smart/Chameleon
  • Current attribution process:
    • For position states imported from external systems: Market Value + Cash Balance
    • For Murex positions: Market Value from Murex + Pass-Cache from Chameleon

Design Challenges

  • Including pass-cache in all runs would require splitting existing runs
  • This could potentially require:
    • Adding ~90 new tasks in controlling (for SA only)
    • Creating ~90 new merge tasks to recombine outputs
    • Reconsidering the entity-set-based design
  • Different system combinations present unique challenges:
    • Osiris positions present special challenges as cash in those portfolios is part of scope
    • For IBD and Corporate Bank positions, both market value and cash balance come from external systems

Proposed Solution Ideas

  • Create one run with two logical sources to handle different position types
  • Alternative approach: Import position sets, generate pass-cash balances from Murex, compare with Chameleon's cash balance, import the difference as corrective position, then run everything including pass-cache
  • This alternative might avoid splitting runs but requires careful implementation and testing

Action Items

  • Meeting leader to share notes and put details in an email
  • Continue refining approach before the upcoming design session for RISC
  • Investigate if the Osiris physical commodity positions are in the same portfolio as their derivatives
